## 1. О системе

**MAX AI Dashboard** — это автоматизированная система, которая:

1. **Собирает** публикации из каналов мессенджера MAX (жалобы жителей на ЖКХ, дороги, благоустройство и т.д.)
2. **Анализирует** каждый пост с помощью искусственного интеллекта (LLM через OpenRouter)
3. **Классифицирует** инциденты по категориям и уровню срочности
4. **Отображает** результат в веб-дашборде с фильтрацией и управлением

**Для кого:** сотрудники городской администрации, ответственные за мониторинг и реагирование на жалобы жителей.

**Результат работы:** вместо ручного просмотра десятков каналов — готовая таблица с приоритизированными инцидентами, готовыми к передаче в профильные службы.

## 2. Архитектура

Система состоит из 4 компонентов, работающих как независимые процессы:

```
  PARSER               AI-АНАЛИЗАТОР            DASHBOARD SERVER         DASHBOARD CLIENT
(Python, polling)  ─────────────────────►  (FastAPI + LLM)     ◄────────► (Express + SQLite)  ◄────────► (React + Vite)
  (webhook)                                Порт 8000                Порт 5000                Порт 3000
```

| Компонент | Назначение | Порт |
|---|---|---|
| **Parser** | Подключается к MAX, опрашивает каналы каждые 30 сек, отправляет новые посты на анализ | — |
| **AI-Анализатор** | Принимает посты, анализирует через LLM (OpenRouter), сохраняет инциденты в БД | 8000 |
| **Dashboard Server** | REST API с JWT-авторизацией, читает и удаляет инциденты из общей БД | 5000 |
| **Dashboard Client** | Веб-интерфейс: таблица инцидентов, фильтры, цветовая кодировка | 3000 |

**Базы данных:**
- `messages.db` — сырые сообщения (записывает и читает только парсер)
- `analytics.db` — общая БД: AI записывает, Dashboard читает/удаляет (WAL-режим для безопасного параллельного доступа)

## 3. Установка

### Требования

| Зависимость | Минимальная версия |
|---|---|

cd ../Dashboard/server

cd ../client

## 4. Настройка

### 4.1 Parser — `Parser/.env`

Скопируйте файл конфигурации:

```bash
cd Parser
copy .env.example .env          # Windows
# cp .env.example .env          # Linux/macOS
```

Откройте `.env` и настройте:

| Параметр | Что указать | Пример |
|---|---|---|
| `PHONE` | Номер телефона UserBot-аккаунта в MAX | `+79933283681` |
| `TARGET_CHANNEL_IDS` | ID каналов через запятую (пусто = все) | `-71887474716883,-71409355871569` |
| `ANALYTICS_WEBHOOK_URL` | URL AI-сервера (обычно не менять) | `http://127.0.0.1:8000/webhook/post` |

> **Как узнать ID каналов?** Запустите парсер с пустым `TARGET_CHANNEL_IDS` — в логе появятся все доступные каналы с их ID. Скопируйте нужные и укажите в `.env`.

### 4.2 AI Server — `AI/.env`

```bash
cd AI
copy .env.example .env
```

| Параметр | Что указать | Пример |
|---|---|---|
| `OPENROUTER_API_KEY` | Ключ от OpenRouter (получить на [openrouter.ai/keys](https://openrouter.ai/keys)) | `sk-or-v1-...` |
| `OPENROUTER_MODEL` | LLM-модель | `qwen/qwen3.6-plus:free` (бесплатно) |

**Доступные модели:**

| Модель | Стоимость | Качество |
|---|---|---|
| `qwen/qwen3.6-plus:free` | Бесплатно | Хорошее |
| `google/gemini-2.0-flash` | Дёшево | Хорошее |
| `openai/gpt-4o-mini` | Средне | Отличное |
| `anthropic/claude-3.5-sonnet` | Дорого | Лучшее |

### 4.3 Dashboard Server — `Dashboard/server/.env`

```bash
cd Dashboard/server
copy .env.example .env
```

| Параметр | Что указать | Пример |
|---|---|---|
| `JWT_SECRET` | Случайная строка 32+ символа | Сгенерируйте на [randomkeygen.com](https://randomkeygen.com) |
| `PORT` | Порт сервера (обычно не менять) | `5000` |
| `DB_PATH` | Путь к общей БД (обычно не менять) | `../../AI/analytics.db` |

> ⚠️ **Важно:** Обязательно замените `JWT_SECRET` на уникальную случайную строку! Значение по умолчанию небезопасно.

## 5. Запуск

Компоненты запускаются в **4 отдельных терминалах** в строгом порядке:

### Терминал 1 — AI-Анализатор

```bash
cd AI
python run_server.py
```

Ожидание: сообщение `🚀 Сервер аналитики запущен на 0.0.0.0:8000`

### Терминал 2 — Парсер

```bash
cd Parser
.venv\Scripts\python comment_parser.py
```

Ожидание: сообщение `✅ Авторизация: <Имя> (ID: ...)` и `🔄 Polling запущен`

### Терминал 3 — Dashboard Server

```bash
cd Dashboard/server
npm run dev
```

Ожидание: сообщение `🚀 Dashboard API запущен → http://localhost:5000`

### Терминал 4 — Dashboard Client

```bash
cd Dashboard/client
npm run dev
```

Ожидание: сообщение `Local: http://localhost:3000/`

> **Порядок важен!** Парсер отправляет вебхуки на AI-сервер, поэтому AI должен быть запущен первым. Dashboard читает ту же БД, что и AI — запускается после.

### Остановка

Обратный порядок: Client (Ctrl+C) → Server (Ctrl+C) → Parser (Ctrl+C) → AI (Ctrl+C).

## 7. Работа с дашбордом

### 7.1 Главная страница

Таблица инцидентов содержит колонки:

| Колонка | Описание |
|---|---|
| **ID** | Внутренний номер инцидента |
| **Канал** | Название канала, откуда поступило сообщение |
| **Текст** | Содержание сообщения (обрезано) |
| **Категория** | Тема проблемы (ЖКХ, Дороги и т.д.) |
| **Срочность** | Уровень приоритета: 🔴 high / 🟡 medium / 🟢 low |
| **Статус** | Текущее состояние: `new` / `in_progress` / `resolved` / `ignored` |
| **Дата** | Дата и время публикации |
| **Действие** | Кнопка удаления |

### 7.2 Цветовая кодировка

**Срочность:**

| Цвет | Уровень | Значение |
|---|---|---|
| 🔴 Красный | **high** | Угроза жизни/здоровью, ЧП, критическая инфраструктура |
| 🟡 Жёлтый | **medium** | Влияет на комфорт жителей, требует внимания в течение дней |
| 🟢 Зелёный | **low** | Пожелание, плановая проблема, не срочный вопрос |

**Статус:**

| Цвет | Статус (отображается в таблице) | Значение |
|---|---|---|
| 🔵 Синий | `new` | Новый инцидент, ещё не обработан |
| 🟡 Жёлтый | `in_progress` | В работе — передан в профильную службу |
| 🟢 Зелёный | `resolved` | Решён — проблема устранена |
| ⚪ Серый | `ignored` | Проигнорирован — не требует реакции |

> Статус отображается в таблице на английском (`new`, `in_progress` и т.д.). Изменить статус можно только через API AI-сервера (см. раздел 9.1). Удаление инцидента доступно через кнопку в дашборде.

### 7.3 Фильтрация

Над таблицей расположены фильтры:

- **Категория** — выпадающий список (загружается из БД автоматически)
- **Срочность** — выбор: high / medium / low
- **Сбросить** — очистка всех фильтров, возврат к полному списку

Инциденты всегда отсортированы по срочности (high → medium → low), внутри одной срочности — по дате (новые сверху).

### 7.4 Удаление инцидента

1. Нажмите кнопку **«Удалить»** в строке инцидента
2. Подтвердите действие во всплывающем окне
3. Инцидент немедленно исчезает из таблицы (удаляется из БД и из отображения)

> Перезагрузка страницы **не требуется** — интерфейс обновляется автоматически.

### 7.5 Выход из системы

Нажмите кнопку **«Выйти»** в верхней части страницы. Токен удаляется из localStorage, вы возвращаетесь на страницу входа.

## 8. Категории и срочность

### Категории инцидентов

| Категория | Когда присваивается | Примеры |
|---|---|---|
| 🏠 **ЖКХ** | Водоснабжение, канализация, отопление, электричество, мусор | «В доме 3 дня нет воды», «Прорвало трубу» |
| 🛣️ **Дороги** | Ямы, тротуары, знаки, светофоры, покрытие | «Огромная яма на Ленина», «Не работает светофор» |
| 🌳 **Благоустройство** | Парки, пляжи, площадки, озеленение, чистота | «Мусор на пляже», «Сломана качель» |
| ⚠️ **Безопасность** | Аварийные здания, упавшие деревья, открытые люки | «Рушится стена», «Открытый люк на детской площадке» |
| 🚨 **ЧП** | ДТП с пострадавшими, пожары, подтопления | «Пожар на складе», «ДТП с пешеходом» |
| 🌿 **Экология** | Загрязнение, свалки, вырубка, качество воздуха | «Несанкционированная свалка», «Чёрная река» |
| 🚌 **Транспорт** | Маршруты, расписания, переполненность | «Автобус ходит раз в 2 часа» |
| 📢 **Обращение к власти** | Прямые вопросы к администрации | «Когда отремонтируют дорогу?», «Планы по стройке» |
| 📋 **Другое** | Всё, что не подходит под категории | Рекламные посты, личные мнения |

### Уровни срочности

| Уровень | Критерий | Пример |
|---|---|---|
| 🔴 **high** | Угроза жизни/здоровью, массовая проблема, критическая инфраструктура | Прорыв трубы, пожар, ДТП, обрушение |
| 🟡 **medium** | Проблема влияет на комфорт жителей, требует внимания в течение дней | Яма на дороге, неработающий фонарь, мусор |
| 🟢 **low** | Пожелание, плановая проблема, не срочный вопрос | Предложение по благоустройству, плановый ремонт |

## 9. API

### 9.1 AI-Анализатор (порт 8000) — без авторизации

| Метод | Эндпоинт | Описание | Параметры |
|---|---|---|---|
| `POST` | `/webhook/post` | Принять пост, AI-анализ | JSON: `{message_id, channel_id, channel_name, text, link, timestamp, date}` |
| `GET` | `/actionable` | Список инцидентов | Query: `status`, `urgency`, `limit` (по умолч. 50) |
| `GET` | `/actionable/{message_id}` | Конкретный инцидент | Path: `message_id` |
| `PATCH` | `/actionable/{message_id}/status` | Обновить статус | Query: `status` (new / in_progress / resolved / ignored) |
| `GET` | `/stats` | Статистика | — |
| `GET` | `/health` | Проверка работоспособности | — |

**Пример — обновление статуса:**

```bash
curl -X PATCH "http://127.0.0.1:8000/actionable/42/status?status=in_progress"
```

**Пример — статистика:**

```bash
curl http://127.0.0.1:8000/stats
```

```json
{
  "total_actionable": 156,
  "by_urgency": { "high": 12, "medium": 89, "low": 55 },
  "by_status": { "new": 45, "in_progress": 31, "resolved": 70, "ignored": 10 },
  "by_category": { "ЖКХ": 67, "Дороги": 34, "Благоустройство": 28 }
}
```

### 9.2 Dashboard Server (порт 5000)

| Метод | Эндпоинт | Авторизация | Описание |
|---|---|---|---|
| `POST` | `/api/auth/register` | Нет | Регистрация нового пользователя |
| `POST` | `/api/auth/login` | Нет | Вход, получение JWT-токена |
| `GET` | `/api/posts` | JWT | Инциденты с фильтрацией |
| `DELETE` | `/api/posts/:id` | JWT | Удалить инцидент |
| `GET` | `/api/posts/categories` | JWT | Список категорий |
| `GET` | `/api/health` | Нет | Проверка работоспособности |

**Пример — регистрация:**

```bash
curl -X POST http://localhost:5000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username": "operator_zkh", "password": "secure_pass_123"}'
```

**Пример — вход:**

```bash
curl -X POST http://localhost:5000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "admin"}'
```

Ответ содержит `token`. Используйте его в заголовке `Authorization: Bearer <token>`.

**Пример — запрос с токеном:**

```bash
curl http://localhost:5000/api/posts \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."
```

## 10. Устранение проблем

### Парсер не подключается к MAX

**Симптом:** ошибка авторизации в логе парсера.

**Решение:**
- Проверьте номер телефона в `PHONE` — он должен быть зарегистрирован в MAX
- При первом входе может потребоваться подтверждение сессии (код из SMS)
- Убедитесь, что пакет `maxapi-python` установлен: `pip list | grep maxapi`

### Парсер не находит каналы

**Симптом:** в логе «❌ Нет каналов для парсинга!»

**Решение:**
- Оставьте `TARGET_CHANNEL_IDS` пустым — парсер выведет все доступные каналы
- Скопируйте нужные ID и укажите в `.env`
- Убедитесь, что UserBot-аккаунт состоит в этих каналах

### AI не анализирует посты

**Симптом:** в логе AI-сервера ошибки OpenRouter.

**Решение:**
- Проверьте `OPENROUTER_API_KEY` в `AI/.env` — он должен начинаться с `sk-or-v1-`
- Проверьте интернет-соединение
- Посетите [openrouter.ai](https://openrouter.ai) — возможно, сервис недоступен
- Для отладки включите `LOG_LEVEL=DEBUG` в `AI/.env`

### Dashboard не загружается

**Симптом:** страница логина не открывается или ошибка 401/403.

**Решение:**
- Убедитесь, что Dashboard Server запущен: `http://localhost:5000/api/health` должен вернуть `{"status":"ok"}`
- Убедитесь, что Dashboard Client запущен: `http://localhost:3000`
- Проверьте `JWT_SECRET` в `.env` — он должен быть задан
- Попробуйте войти как `admin / admin`

### Токен истёк

**Симптом:** автоматический выход из дашборда.

**Решение:** JWT-токен действует 24 часа. Войдите заново.

### Инциденты не появляются

**Симптом:** дашборд показывает пустую таблицу, но в каналах есть публикации.

**Решение:**
- Проверьте, что парсер работает (лог polling-цикла)
- Проверьте, что AI-сервер принимает вебхуки: `http://127.0.0.1:8000/health`
- Проверьте логи AI-сервера — возможны ошибки LLM
- Обратите внимание: AI сохраняет только посты с `requires_response=true`. Посты без проблем (новости, реклама) пропускаются — это нормальное поведение

### Ошибка при удалении инцидента

**Симптом:** ошибка 404 или сообщение об ошибке при удалении.

**Решение:**
- Инцидент мог быть уже удалён (например, через API AI-сервера `DELETE /api/posts/:id`)
- Ошибка отобразится во всплывающем окне — закройте его и обновите данные
- Проверьте подключение к Dashboard Server: `curl http://localhost:5000/api/health`
